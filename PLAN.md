# План разработки tg_mini_app

Документ объединяет **исходное ТЗ заказчика** (зафиксировано в переписке: [ТЗ суши Mini App](e2acd732-678f-4cad-b901-b55ce839f761)) и **фактическое состояние кода** на момент составления. Его можно править по мере согласований с заказчиком.

---

## 1. Цель продукта

Telegram Mini App для заказа суши с UX, близким к агрегаторам доставки еды, плюс бот для согласования и оплаты, плюс веб-панель учёта. Сервер сначала локально, затем перенос в облако.

---

## 2. Исходное ТЗ (краткая выжимка)

| Область | Что просили |
|--------|-------------|
| **Шапка Mini App** | Время работы; раздел «Помощь» — чтобы клиент мог сразу написать с вопросом |
| **Категории** | В т.ч. «Заказывали ранее», «Суши», «Роллы», «Сеты» (пример из ТЗ) |
| **Каталог** | Плитка с фото; кнопка «+» внизу справа на карточке |
| **Корзина** | Появляется снизу при добавлении (у нас — FAB справа снизу) |
| **Оформление** | Адрес, время доставки; кнопка «Согласовать заказ» |
| **Согласование** | Оператору в бота: сводка + «Да» / «Нет» / «Изменить» → при «Изменить» текст правок клиенту + «Да»/«Нет» у клиента |
| **Оплата** | После согласования: «Наличные» / «Карта»; наличные → уведомление оператору, заказ активен; карта — реальная оплата, после успеха — уведомление и активный заказ |
| **Веб-панель** | Учёт и управление: товары (фото, категория, название, вес, цена, описание, состав); категории; отметка выполнения заказов |
| **Дополнительно** | Заказ через **inline** в чате; **общая корзина** (поделиться, дополняют другие); **команды в боте** для управления |
| **Инфра** | Сначала ПК для отладки, потом облако |

**Уже согласованное отклонение от примера категорий в ТЗ:** в сиде используются три раздела — «Горячие жаренные ролы», «Запеченые ролы», «Холодные ролы» (без отдельных «Макидзуси / Хосомаки / Футомаки»). При необходимости структуру меню возвращаем к формулировкам заказчика.

## 3. Текущее состояние (по коду)

### 3.1. Сделано и рабочее

- **Backend (FastAPI):** каталог (`/catalog/*`), корзина, заказы, проверка `initData` Mini App, режим `APP_ENV=local` для отладки без Telegram.
- **БД (SQLite):** категории, товары, корзина с `shared_key`, заказы, статусы в духе конечного автомата (`order_flow.py`).
- **Бот (aiogram):** `/start`, кнопка меню / Web App, цепочка согласования (Да/Нет/Изменить), ввод правок оператором, ответ клиента, выбор наличные/карта (**карта — заглушка**), команды оператора `/ship`, `/delivered`, настройка меню и команд при старте.
- **Mini App:** витрина, тема Telegram, сетка товаров, корзина-drawer, оформление (адрес, время), отмена до согласования, опрос статуса заказа.
- **Панель оператора:** список заказов, смена статусов, согласование из панели (частично дублирует бота).
- **API общей корзины:** `shared_key`, эндпоинты вида «получить корзину по share» (см. `cart.py`) — **UI «поделиться» в Mini App не подключён**.

### 3.2. Сделано частично или иначе, чем в ТЗ

| Тема | Статус |
|------|--------|
| Время работы / помощь в шапке | Вёрстка есть (`webapp.html`); ссылка «Помощь» в **app_v3.js не привязана** (в отличие от старого `app.js`) — нужно восстановить сценарий (tg.openTelegramLink на оператора / deep link) |
| Категории как в ТЗ | Сейчас 3 категории из сида; нет вкладки **«Заказывали ранее»** (нужны история заказов + API) |
| Плитка «+» на фото | Нужна сверка с макетом; возможны отличия от «как у Еды» |
| Оплата картой | **Заглушка**; нужны **Telegram Payments** + `PROVIDER_TOKEN` и обработка `pre_checkout` / `successful_payment` |
| Веб-панель | Есть операторская часть по заказам; **полного CRUD каталога** (фото, состав, категории) в панели нет — данные в основном через сид/БД |
| Inline-заказ в чате | **Не реализовано** |
| Общая корзина для друзей | **Backend готов**, нет сценария в UI и deep link `startapp` |
| Миграции БД | Таблицы через `create_all`; **Alembic в репо не подключён** — для продакшена желательно ввести |
| Облако | Обсуждалось отдельно; **не внедрялось** |

## 4. Предлагаемые фазы (дорожная карта)

Нумерация — логический порядок; параллельно можно закрывать UX и админку.

### Фаза A — Закрыть расхождения с ТЗ по клиентскому приложению

1. Починить **«Помощь»** в `app_v3.js` (и при необходимости вынести время работы в настройки/API).
2. Решить с заказчиком финальный список **категорий**; при согласии — добавить вкладку **«Заказывали ранее»** (API последних заказов пользователя + отдельная выдача/быстрое добавление).
3. Довести **карточки и FAB** до согласованного с заказчиком вида («как Яндекс Еда»).
4. **Поделиться корзиной:** кнопка в Mini App, копирование ссылки, открытие по `start_param` / присоединение к `shared_key`, политика конфликтов (кто может менять корзину).

### Фаза B — Оплата и надёжность заказа

1. Внедрить **реальную оплату картой** (Telegram Invoice + провайдер).
2. Журналирование ключевых событий заказа (опционально таблица событий) для споров и отладки.
3. Подключить **Alembic** и стратегию миграций перед выкладкой в облако.

### Фаза C — Веб-панель (учёт и каталог)

1. CRUD **категорий** (создание / переименование / порядок / скрытие).
2. CRUD **товаров** с загрузкой **фото**, полями состав, вес, цена, описание, привязка к категории.
3. Список заказов с фильтрами и явной **отметкой «выполнен»** (если не покрывается текущими статусами — уточнить с заказчиком).

### Фаза D — Расширенный Telegram

1. **Inline-режим:** выдача позиций в чат, добавление в корзину / открытие Mini App.
2. Расширение **команд бота** под операционные сценарии заказчика (без дублирования панели там, где неудобно).

### Фаза E — Продакшен

1. **Деплой** (VPS / облако), HTTPS, `BASE_URL`, секреты, бэкапы БД.
2. Мониторинг ошибок и нагрузки (минимум — логи + healthcheck).

## 5. Открытые вопросы к заказчику

- Финальные названия и состав **категорий** (остаёмся на трёх «роллах» или возвращаемся к «суши / роллы / сеты» + «заказывали ранее»).
- Куда ведёт **«Помощь»**: только чат с оператором, или ещё FAQ / телефон?
- **Платёжный провайдер** для Telegram Payments (страна, валюта).
- Нужна ли **одна общая корзина на N человек** с одновременным редактированием или достаточно «пригласить по ссылке и дополнить один раз»?
- Требования к **ролям** в веб-панели (один оператор = админ или отдельный «кухня/курьер»).
